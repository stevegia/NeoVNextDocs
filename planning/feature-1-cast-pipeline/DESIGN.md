# Feature 1 -- Cast Member Pipeline (End-to-End)

## Status Quo

The cast extraction pipeline is **substantially built** but has gaps between the backend pipeline completing and the frontend rendering useful cast data.

### What Already Works

- **Phase 1 handlers** (face_extraction, transcription, diarization): Fully implemented, write to cluster Postgres via `ClusterDbClient`.
- **Phase 2 handler** (face_clustering): Fully implemented. Groups face detections by cosine similarity, creates `persons` records with face embeddings, writes `person_appearances` rows.
- **Phase 3 handler** (voice_face_association): Fully implemented. Links speakers to persons via temporal overlap, sets `voice_sample_path` and `face_thumbnail` on person records.
- **Pipeline orchestration**: `PipelineService` + `PipelineAdvancementService` advance phases 1->2 automatically. Phase 2 completion marks pipeline as "completed".
- **Frontend CastPage**: Has video selector, pipeline progress display, `CastGallery` grid, `PersonDetailPanel` with rename, voice sample playback.
- **PersonsController**: GET by media, GET by id, PUT rename, GET thumbnail (serves file from disk), GET voice-sample.
- **Frontend API client**: `PersonRecord` type, `getPersonsByMedia`, `updatePerson`, `extractCast`, `getPipelineStatus`.

### What Is Missing or Broken

1. **Phase 3 (voice_face_association) is not wired into the pipeline.** The `PipelineService.CheckAndAdvancePipelineAsync` only handles phase 1->2 and 2->completed. There is no phase 3 advancement. Voice-face association never runs automatically.

2. **Persons table schema mismatch.** The cluster Postgres `persons` table (migration 009) has columns: `person_id, media_id, display_name, face_thumbnail_path, voice_sample_path, appearance_count`. But the Python `ClusterDbClient.create_person()` writes to columns: `person_id, name, face_thumbnail, voice_sample_path, face_embedding, voice_embedding, auto_generated`. The migration schema does NOT have `face_embedding`, `voice_embedding`, `auto_generated`, or `name` columns. The `PersonsController` reads `display_name` from the DB. **This will crash at runtime.**

3. **No `person_appearances` table in any migration.** The `face_clustering` handler calls `_insert_person_appearance` which writes to `person_appearances`, but no migration creates this table.

4. **face_thumbnail resolution is broken.** The `voice_face_association` handler sets `face_thumbnail` to `face_detection:{detection_id}` -- a reference string. The `PersonsController.GetThumbnail` tries to serve this as a physical file path, which will fail. The actual face crop JPEGs are written by `face_extraction` into `{output_dir}/face_thumbnails/face_f{frame}_{idx}.jpg` but these paths are in the worker's output dir, not referenced anywhere the API can find them.

5. **No cast member portrait generation.** The user wants ComfyUI-generated portrait images for each person based on visual descriptions. This requires: (a) getting a description of the person (via visual tagger or LLM), (b) submitting to ComfyUI as a t2v/t2i job, (c) storing the result.

6. **Cast member description generation.** No mechanism exists to describe a person from their face crops. The visual tagger (`visual_tagging.py`) uses ImageNet-1k labels -- not useful for person descriptions. An LLM with vision (Feature 3) would be far better. We need an `ILlmService` interface to call for descriptions.

7. **No face crops served via API.** Face crop thumbnails are generated by `face_extraction` but stored in the worker's output directory. They need to be synced to the API layer or served through the existing artifact/cache system.

## Implementation Plan

### 1. Fix Cluster Postgres Schema (Migration 010)

Create `src/data/migrations/cluster/010_fix_persons_schema.sql`:

```sql
-- Fix persons table to match what the Python handlers actually write
ALTER TABLE persons ADD COLUMN IF NOT EXISTS name TEXT;
ALTER TABLE persons ADD COLUMN IF NOT EXISTS face_thumbnail TEXT;
ALTER TABLE persons ADD COLUMN IF NOT EXISTS face_embedding vector(512);
ALTER TABLE persons ADD COLUMN IF NOT EXISTS voice_embedding vector(192);
ALTER TABLE persons ADD COLUMN IF NOT EXISTS auto_generated BOOLEAN NOT NULL DEFAULT TRUE;

-- Backfill: copy display_name -> name for any existing rows
UPDATE persons SET name = display_name WHERE name IS NULL AND display_name IS NOT NULL;

-- Add description column for LLM-generated person descriptions
ALTER TABLE persons ADD COLUMN IF NOT EXISTS description TEXT;

-- Add elevenlabs_voice_id for Feature 3 integration
ALTER TABLE persons ADD COLUMN IF NOT EXISTS elevenlabs_voice_id TEXT;

-- Add generated_portrait_path for ComfyUI portrait images
ALTER TABLE persons ADD COLUMN IF NOT EXISTS generated_portrait_path TEXT;

-- Create person_appearances table (referenced by face_clustering handler)
CREATE TABLE IF NOT EXISTS person_appearances (
    appearance_id   TEXT PRIMARY KEY NOT NULL,
    person_id       TEXT NOT NULL,
    media_id        TEXT NOT NULL,
    first_seen_sec  REAL NOT NULL DEFAULT 0,
    last_seen_sec   REAL NOT NULL DEFAULT 0,
    detection_count INTEGER NOT NULL DEFAULT 0,
    source          TEXT NOT NULL DEFAULT 'face',
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_person_appearances_person ON person_appearances(person_id);
CREATE INDEX IF NOT EXISTS idx_person_appearances_media ON person_appearances(media_id);

-- face_detections: add face_thumbnail_path for serving crops
ALTER TABLE face_detections ADD COLUMN IF NOT EXISTS face_thumbnail_path TEXT;
```

### 2. Wire Phase 3 into Pipeline Advancement

Modify `PipelineService.CheckAndAdvancePipelineAsync`:
- Phase 1 complete -> enqueue `face_clustering` (phase 2) -- already works
- Phase 2 complete -> enqueue `voice_face_association` (phase 3) -- **NEW**
- Phase 3 complete -> mark pipeline as `completed` -- **NEW**

### 3. Fix Face Thumbnail Resolution Chain

**In face_extraction handler:** After writing face crop to disk, store the crop path in the output JSON per-face entry (already does this as `face_thumbnail_path`).

**In the Go supervisor / result ingestion:** When face_extraction completes, the C# API ingests artifacts. Extend artifact ingestion to also write `face_thumbnail_path` to the `face_detections` row in Postgres.

**In voice_face_association handler:** Instead of setting `face_thumbnail` to `face_detection:{id}`, resolve the actual `face_thumbnail_path` from the detection row and set it on the person.

**In PersonsController:** The `GetThumbnail` endpoint should look up `face_thumbnail_path` from the person record and serve it. If the path is a worker output path, it needs to be accessible. Since we're on RunPod with shared volumes, the path should be directly accessible.

### 4. Fix PersonsController Column Mapping

Update the SELECT query to use the actual column names that exist after migration 010:
- `name` instead of `display_name` (or alias both)
- `face_thumbnail` instead of `face_thumbnail_path`
- Add `description`, `elevenlabs_voice_id`, `generated_portrait_path` to the response

### 5. Person Description Generation (Integration Point with Feature 3)

Define an `ILlmService` interface (implemented by Feature 3) with:
```csharp
public interface ILlmService
{
    Task<string?> DescribePersonFromImageAsync(string imagePath, CancellationToken ct = default);
    Task<string?> GenerateTextAsync(string prompt, CancellationToken ct = default);
}
```

After voice_face_association completes (or as a phase 4), call `ILlmService.DescribePersonFromImageAsync` with the person's best face crop to generate a natural-language description. Store it in `persons.description`.

If `ILlmService` is not configured (no API keys), skip this step gracefully.

### 6. Portrait Generation via ComfyUI (Optional Phase 5)

After a person has a `description`, optionally enqueue a ComfyUI text-to-image job using the description as prompt. The resulting image becomes `persons.generated_portrait_path`.

This is a nice-to-have and can be triggered manually from the frontend rather than automatically in the pipeline.

### 7. Frontend Enhancements

- **PersonCard**: Show `generated_portrait_path` image if available, fall back to `face_thumbnail`.
- **PersonDetailPanel**: Add description display, voice clone status, portrait generation button.
- **CastPage**: Add a "Generate Descriptions" bulk action button (calls LLM for all persons without descriptions).

### 8. API Endpoint Additions

```
GET  /api/persons/{id}/face-crops     -- returns list of face detection crops for a person
POST /api/persons/{id}/generate-description  -- calls ILlmService to describe person
POST /api/persons/{id}/generate-portrait     -- queues ComfyUI t2i job for portrait
GET  /api/persons/{id}/portrait       -- serves generated portrait image
```
